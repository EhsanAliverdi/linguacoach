# Adaptive Curriculum Sprint 5 — AI Composer (Today/Practice Gym Selection Cutover)

**Date:** 2026-07-20
**Related sprint / feature:** Sprint 5 of the multi-sprint Adaptive Curriculum initiative — see `docs/architecture/adaptive-curriculum-skill-graph.md`, and the Sprint 1-4 review docs for the skill graph, content re-tagging, goal vector, and node mastery this sprint's composer reasons over. This sprint replaces Today/Practice Gym's mechanical selection heuristic with an AI composer — the first sprint to hard-cutover a real, currently-executing selection algorithm rather than only add new signals alongside it.

## Context / decision

Per the target architecture, delivery selection should reason jointly over skill-graph mastery gaps, goal-vector relevance, spacing, and novelty — replacing the flat CEFR+tag `ScoreModule` heuristic `TodayPlanModuleSelectionService`/`PracticeGymModuleSelectionService` used. Research before writing any code (via a dedicated Explore pass) surfaced one critical finding that reshaped this sprint's scope, resolved via an explicit user decision:

**`CurriculumObjective`/`CurriculumRoutingService` cannot be fully deleted this sprint.** `LearningPlanService.BuildObjectiveSequenceAsync` calls `CurriculumRoutingService.RecommendAsync` on the hot path of every plan generation, and `LearningPlanService.TryUpdateObjectiveProgressAsync` (triggered on every real activity submit) still depends on `EvaluateObjectiveMasteryAsync`'s legacy objective-key grouping. Live DB state confirmed this is genuinely load-bearing, not dead: 5 active `student_learning_plans`, 10 `student_learning_plan_objectives`, 34 `curriculum_objectives`. Deleting the subsystem outright without first replacing plan-generation sequencing would break Journey/Progress/Dashboard for every student with a plan — a second full sprint's worth of work in its own right.

- Decision (AskUserQuestion, recommended option chosen): **narrow Sprint 5's scope** — build the AI composer for Today/Gym selection (a hard cutover, since that algorithm genuinely is being replaced), and retire only the two branches Sprint 4 already proved permanently dead (`CurriculumRoutingService`'s mastery-exclusion, `StudentMasteryEvaluationJob`'s mark-mastered/mark-completed no-ops). Leave `CurriculumRoutingService.RecommendAsync`, `CurriculumObjective`, `AdminCurriculumController`, and `Module.ObjectiveKey` (a small, genuinely still-working self-directed Practice Gym filter, confirmed reachable via the admin Module-authoring UI, not dead) untouched. Full retirement is pushed to a new Sprint 6.

Separately, research confirmed the content pool is currently **fully empty for delivery**: 0 Modules satisfy `ModuleEligibility.AvailableForNewStudentDeliveryExpr` (both approved Modules are archived; the two non-archived Modules are still `PendingReview`). Today/Gym selection was already hitting 100% fallback in live dev before this sprint, for content-authoring reasons unrelated to the selection algorithm itself.

## Files reviewed

`TodayPlanModuleSelectionService.cs`/`PracticeGymModuleSelectionService.cs` (full algorithms — eligibility query, Lesson/Exercise approval check, 14-day reuse cooldown, CEFR filter/broadening, the `ScoreModule` heuristic being replaced, the skill-diversity greedy-selection guard), their paired `*SelectionContracts.cs` DTOs, `SessionQueryHandler`/`PracticeGymSuggestionService` (the two real production call sites — confirmed admin preview controllers bypass the recorder and are read-only diagnostics, not a second production path), `ModuleSkillGraphTaggingService.cs`/`SkillGraphDraftingService.cs` (the AI-draft-then-validate convention this sprint's composer follows exactly), `AiExecutionService.cs` (`ExecuteWithMetaAsync`, `QuotaCheckedFeatures`, `ResolveLlmCategory` — confirmed the new feature key is unmetered against student quota like Sprint 1/2's advisory keys, correctly since Today/Gym are core free flows), `DefaultAiSeeder.cs` (prompt registration/seeding convention), `CurriculumRoutingService.cs` (`FilterByMastered` and its Rule-5 mirror — both proven permanently dead by Sprint 4's key-space change), `StudentMasteryEvaluationJob.cs` (the mark-mastered/mark-completed no-op loop), `LearningPlanService.cs` (`BuildObjectiveSequenceAsync`, `TryUpdateObjectiveProgressAsync`, `EvaluateObjectiveMasteryAsync`'s one caller — confirmed why full retirement isn't safe this sprint), `Module.ObjectiveKey`'s consumers (`admin-module` frontend, `PracticeGymModuleSelectionService`'s self-directed narrowing — confirmed still a real, reachable feature).

## What was built

### Backend — AI composer

- **`ICurriculumComposerService`/`ComposerCandidate`/`ComposerRankingRequest`/`ComposerRankingResult`** (`LinguaCoach.Application.Composer`) — the caller (a selector) still owns eligibility: Approved/non-archived, has an approved Lesson+Exercise, CEFR match, the 14-day reuse-cooldown. The composer only ranks among an already-eligible pool. Each `ComposerCandidate` carries two real, deterministic facts the caller computes — never inferred by the AI itself: `IsWeaknessMatch` (the candidate's Module covers a skill-graph node the student is Weak/AtRisk on, resolved via `StudentMasteryEvaluationService.EvaluateStudentAsync`'s Sprint-4 node grouping, joined through `ModuleSkillGraphNodeLink`) and `IsGoalMatch` (the candidate's context tags overlap the student's top-weighted `StudentGoalWeight` tags, weight ≥ 0.5). A third signal, `RecentlyPractisedSameSkill`, is a 3-day novelty band computed by the caller from the same assignment-history tables the 14-day cooldown already reads — a soft "you just did this" hint distinct from the hard exclusion.
- **`CurriculumComposerService`** (`LinguaCoach.Infrastructure.Composer`) — structurally identical to `ModuleSkillGraphTaggingService`: one bounded AI call (new prompt `curriculum_composer_rank_candidates`, seeded via `DefaultAiSeeder`), retried once on bad JSON, never throws, every ranked id validated against the real candidate set given in the request before being trusted (a hallucinated id is dropped, never applied). Zero candidates short-circuits to failure without an AI call. Also passes `PreferredSessionLengthMinutes` (Today) / `RequestedDifficulty` (Practice Gym) as prompt context so the AI can reason over session-length/difficulty fit the same way the old heuristic scored it — feature parity for those two signals, not dropped.
- **Both selectors hard-cutover**: `ScoreModule` and the skill-diversity greedy-selection loop are deleted from both `TodayPlanModuleSelectionService` and `PracticeGymModuleSelectionService`. Eligibility/CEFR/recency-cooldown/self-directed-narrowing logic is unchanged. After building the eligible pool, each selector resolves `IsWeaknessMatch`/`IsGoalMatch`/`RecentlyPractisedSameSkill` per candidate, calls `ICurriculumComposerService.RankCandidatesAsync`, and maps the validated ranked ids back to its own result DTO in the composer's order. A composer failure (AI unavailable, no valid candidate returned) degrades to the selector's existing `FallbackRequired=true` path — the same honest-degrade contract these selectors already had, not a new failure mode. No fallback to the deleted `ScoreModule` heuristic exists.
- New `MasteryEvaluationReason.ContentDelivery` enum value, since neither selector's mastery lookup fit any existing reason.
- DI registration in `DependencyInjection.cs`; prompt content instructs the AI explicitly ("You do NOT decide anything final — every id you return is checked against a fixed candidate list before it is ever applied") and gives an explicit prioritization order (requested skill/subskill/objective → weakness match → goal match → skill diversity/novelty → duration/difficulty fit as a tie-breaker), mirroring how a real teacher would triage.

### Backend — proven-dead code removal

- `CurriculumRoutingService.FilterByMastered` (the method) and its call site deleted outright — Sprint 4 made this comparison permanently unreachable (`request.MasteredObjectiveKeys` is sourced from skill-graph node keys, an unrelated key space to `CurriculumObjective.Key`, so it could never match). The inline Rule-5 mirror inside `TrySelectPreferredObjectiveAsync` was simplified the same way.
- `StudentMasteryEvaluationJob`'s per-objective `MarkObjectiveMasteredAsync`/`MarkObjectiveCompletedAsync` sweep loop deleted — same permanently-dead reasoning (every call always no-op'd via `"objective_not_in_plan"`). `planNeedsRefresh`'s count-based regeneration trigger, which is unrelated to this loop, is unchanged.
- `CurriculumRoutingService.RecommendAsync`, `CurriculumObjective`, `AdminCurriculumController`, `admin-curriculum` page, and `Module.ObjectiveKey` are **deliberately untouched** — all four confirmed still real and load-bearing (or, for `Module.ObjectiveKey`, still a small genuinely-working feature) per the narrowed-scope decision. Full retirement is Sprint 6's job.

### Tests

- Unit tests for both selectors reworked (not just patched): removed 5 tests that asserted the deleted `ScoreModule` heuristic's specific priority ordering (context/focus tag match, session-length proximity, difficulty proximity, skill-match-causing-selection), replaced with tests that verify the actual new integration — `RequestedSkill`/`PreferredSessionLengthMinutes`/`RequestedDifficulty` correctly reach the composer request, `IsGoalMatch`/`IsWeaknessMatch` are correctly computed from real seeded `StudentGoalWeight`/`ModuleSkillGraphNodeLink`+`SkillGraphNode` rows (not mocked), the composer's ranked order is honored end-to-end, and a composer failure degrades to `FallbackRequired=true` rather than throwing. New `FakeCurriculumComposerService`/`FakeStudentMasteryEvaluationService` test doubles (pass-through by default, programmable ranking override, force-failure switch) — never a real AI call, per this repo's "tests use fake providers" convention.
- 4 `CurriculumRoutingServiceTests`/`CurriculumRoutingServicePreferredObjectiveTests` tests reworked to document the new (no-exclusion) behavior instead of asserting the removed mastery-exclusion — same discipline as Sprint 4's test rework, not deleted silently.
- **Integration tests**: 2 `TodayPlanModulePipelineEndpointTests` failed after the cutover, because the integration test host has no real AI provider credentials configured, so every composer call legitimately failed and every selection fell back — correct behavior, but it meant these tests (and any future ones) couldn't exercise a real selection end-to-end. Fixed by registering a `FakeCurriculumComposerService` in `ApiTestFactory`, mirroring the existing `FakeFileStorageService`/`FakeAudioDurationProbe` convention for external dependencies the test/CI environment doesn't have real credentials for.

## Decisions made

1. Narrowed Sprint 5 scope: build the composer and hard-cutover Today/Gym selection now; defer full `CurriculumObjective`/`CurriculumRoutingService` retirement to a new Sprint 6, since it requires replacing `LearningPlanService`'s plan-generation sequencing — real work against live student plan data, not a same-sprint deletion.
2. Delete only the two branches Sprint 4 already proved permanently dead (`CurriculumRoutingService`'s mastery-exclusion, `StudentMasteryEvaluationJob`'s no-op sweep) plus their now-obsolete test assertions — everything else in the legacy curriculum system stays until Sprint 6.
3. `Module.ObjectiveKey` left untouched — confirmed still a real, reachable, working feature (admin Module authoring, Practice Gym self-directed narrowing), not dead code, so retiring it doesn't belong in a "remove proven-dead code" pass.
4. Composer candidates carry deterministic, caller-computed facts (`IsWeaknessMatch`/`IsGoalMatch`/`RecentlyPractisedSameSkill`) rather than raw mastery/goal data — the AI reasons over real facts, it never computes them itself, matching the AI-draft-then-validate discipline's spirit even though this isn't a draft-and-approve flow.
5. A composer failure degrades to the selector's existing `FallbackRequired=true` contract, not a new failure mode — consistent with these selectors' pre-existing "never throws to the caller" guarantee.

## Risks / unresolved questions

- Real end-to-end verification (a real student getting a real AI-ranked Today/Gym selection) remains blocked on the same content-thinness gap Sprint 2 first flagged — not a defect in this sprint's code, but still genuinely unverified against live traffic.
- Every Today/Gym request that reaches the composer now costs one AI call (unlike the fully-deterministic pre-Sprint-5 selectors) — acceptable at current near-zero content volume, but worth watching once Sprint 6 seeds real content and real request volume begins; the prompt is deliberately small (candidates capped at 40, `MaxOutputTokens=700`) to keep this bounded.
- `CurriculumRoutingService`/`CurriculumObjective`/`AdminCurriculumController`/`Module.ObjectiveKey` remain live, un-retired systems running alongside the new composer — not a "two systems doing the same job" conflict (they serve different consumers: plan-generation sequencing vs. Today/Gym content selection), but a real ongoing maintenance surface until Sprint 6 closes it out.

## Final verdict

Sprint 5 complete and verified: 0 backend build errors, 3,934/3,934 tests passing (including reworked tests seeding real FK-validated Module/SkillGraphNode/StudentGoalWeight chains rather than mocking the new signals), 0 new frontend build errors, live deployment confirmed healthy via a forced no-cache rebuild with the new AI prompt confirmed seeded and active. The hard cutover is real — `ScoreModule` no longer exists in either selector — and the two mastery-exclusion branches Sprint 4 proved permanently dead were deleted, not just documented. Full `CurriculumObjective` system retirement was deliberately descoped to Sprint 6 after research showed it's still load-bearing for live student learning plans.
