## Findings

### Priority: Critical (must verify)

**1. Existing placement flow untouched.** The original `PlacementAssessment` entity retains all existing fields, constructor, and methods (`Start()`, `AdvanceSection()`, `Complete()`). The new `CreateAdaptive()` factory method is additive. `PlacementService` (the old handler-based service) is unchanged. No regressions.

**2. Clean architecture boundaries respected.** Domain has no EF Core or ASP.NET dependencies. Application defines interfaces only. Infrastructure implements. API depends on Application interfaces. ArchTest suite (3 tests) continues to pass.

**3. Migration T62 is hand-authored.** It was not generated by `dotnet ef migrations add` because the model snapshot would require a running provider. The hand-authored migration matches the EF configuration exactly. SQLite in-memory tests run `EnsureCreated` (not migrations), so the snapshot is not required for tests.

### Priority: High

**4. Deterministic item bank (72 items).** The seeded item bank covers 6 skills × 4 CEFR levels × 3 items = 72 items total. Items are simple MCQ/gap-fill with a known `CorrectAnswer` field. Scoring is fully deterministic — no AI dependency. This satisfies Phase 13A scope. AI-generated adaptive items are deferred to Phase 13B+.

**5. Confidence algorithm.** `confidence = min(itemsEvaluated / 6.0, correctRatio)`. The divisor 6 represents the target evidence count per skill (2 items × 3 levels). This is intentionally conservative. `ConfidenceThreshold` is configurable (default 0.75). Items below the threshold mark the result as provisional.

**6. Learning Plan integration.** `CompleteAssessmentAsync` calls `ILearningPlanService.RegeneratePlanAsync("placement_completed")`. Failure is caught, logged as warning, and surfaced in `LearningPlanRegenerationWarning` DTO field. Placement completion never fails because of plan regeneration failure.

**7. StudentProfile.CefrLevel update.** `AdminSetCefrLevel` is called only when `overallConfidence >= 0.6`. The 0.6 threshold is deliberately lower than the provisional threshold (0.75) — a confirmed-but-uncertain level is still better than no level. This can be revisited in 13B.

### Priority: Medium

**8. Foundation simulation in CompleteAssessmentAsync.** For Phase 13A, items without responses are auto-scored at 70% correct (deterministic by item order: `itemOrder % 10 < 7`). This is a placeholder for the real student response submission endpoint (Phase 13B). The simulation produces stable, testable results.

**9. Admin UI — read-only.** The spec explicitly excludes Start/Complete buttons from the Angular UI. The component shows placement status, CEFR level, confidence, provisional warning, skill breakdown, and last completed date. Empty state is displayed when no placement exists.

**10. Source field.** The `Source` property is set at assessment creation time ("admin", "student"). It is stored but not yet used for access control in Phase 13A.

### Priority: Low

**11. No student-facing placement endpoints.** The `AdminPlacementController` is admin-only. Student-triggered adaptive placement requires a student endpoint (Phase 13B scope).

**12. `Failed = 5` enum value.** Added to `PlacementStatus` per spec but no domain method transitions to `Failed` in Phase 13A. Reserved for future AI evaluation failure handling.

## Decisions Made

1. Used `CreateAdaptive()` static factory method on `PlacementAssessment` instead of a second public constructor. Reason: the existing public constructor requires `firstSectionKey`; a factory avoids ambiguity and sets `IsAdaptive = true` at construction.

2. `CorrectAnswer` stored on `PlacementAssessmentItem`. Reason: deterministic scoring needs to compare student response to the correct answer without re-loading the item bank.

3. Migration T62 is hand-authored (not `dotnet ef migrations add`). Reason: avoids the need for a running PostgreSQL instance in CI. SQLite in-memory tests use `EnsureCreated` which reads EF configurations directly, so the migration is only needed for PostgreSQL production upgrades.

4. Overall CEFR is the minimum of per-skill estimates (conservative). Reason: curriculum routing should never over-estimate the student's level; underestimation leads to review items which is less harmful than over-estimation.

## Risks and Unresolved Questions

1. **Migration T62 not applied to production yet.** It will need `dotnet ef database update` or Docker compose migration runner on next deploy.

2. **Simulation in CompleteAssessmentAsync.** The `itemOrder % 10 < 7` scoring is a placeholder. When Phase 13B lands, real student responses replace this. Tests for this path will need updating.

3. **No expiry job.** The `Expire()` method exists but no background job calls it. Assessments will remain `InProgress` indefinitely until Phase 13B adds the expiry sweep.

4. **Angular admin-student-detail.** The placement section was added as a read-only card. It depends on `getStudentLatestPlacement()` being called during component init. The exact wiring to the component's `ngOnInit` should be reviewed in a live QA pass.
