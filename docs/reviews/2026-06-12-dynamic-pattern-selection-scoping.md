# Dynamic Pattern Selection — Scoping Pass

Date: 2026-06-12
Related: `docs/sprints/current-sprint.md` "Next recommended work" #1; deferred item
"Dynamic pattern selection" in `docs/sprints/current-sprint.md`.
Status: Scoping only — no code changes in this pass

## Context

Today's Lesson is generated by `SessionGeneratorService.GetOrCreateTodaysSessionAsync`
from a fixed per-duration template (`SessionDurationTemplates`), with a single
weak-skill substitution pass. The brief is to make pattern selection genuinely
adaptive: driven by `StudentSkillProfile.ScorePercent` across all skills, CEFR level,
session duration, and repetition history — not just a binary weak/not-weak swap on
one slot.

## Files reviewed

- `src/LinguaCoach.Infrastructure/Sessions/SessionGeneratorService.cs` (full
  `GetOrCreateTodaysSessionAsync` flow, `ApplyWeakSkillSubstitution`)
- `src/LinguaCoach.Infrastructure/Sessions/SessionDurationTemplates.cs` (all 4
  duration templates, `ExerciseStepTemplate` record)
- `src/LinguaCoach.Infrastructure/Activity/PatternSkillUpdateService.cs`
  (`PatternKeyToSkillKey` mapping, `ScorePercent` updates)
- `src/LinguaCoach.Domain/Entities/StudentSkillProfile.cs` (`ScorePercent`,
  `WeakThreshold`, `IsWeak`)
- `docs/architecture/course-session-learning-model.md` (lines ~218-240, the
  "session generator inputs" wishlist)
- `docs/architecture/student-learning-memory.md`

## Findings

### What already exists

- 4 fixed templates (10/15/20/30 min), each a list of `ExerciseStepTemplate`
  (Order, Kind, PatternKey, PrimarySkill, Instructions, EstimatedMinutes).
- `ApplyWeakSkillSubstitution` (SessionGeneratorService.cs:175-227): picks ONE
  "target skill" (first weak skill from `StudentSkillProfile` where
  `ScorePercent < WeakThreshold`, else student's `SkillFocus`, else none), and:
  - If target is Speaking: swap the `WritingTask` step to
    `spoken_response_from_prompt`.
  - If target is Listening: swap the `ContextInput` step to `listen_and_answer`.
  - Otherwise: template unchanged.
- CEFR level (`profile.CefrLevel`) is read but only passed into AI generation
  context (`ActivityGetHandler`), never used to pick *which* pattern/template.
- No repetition history: `FindTodaysSessionAsync` only prevents duplicate sessions
  on the same calendar day. Nothing looks at which `PatternKey`s ran in the last
  N sessions.
- `PatternSkillUpdateService.PatternKeyToSkillKey` maps each pattern to one skill
  key — this is the same mapping Phase 3 P1 reused for AI feedback context, and is
  the natural lookup table for skill-aware pattern selection too.

### Gaps vs. the "Dynamic Pattern Selection" brief

1. **Only one skill substitution per session.** A student weak in both Speaking
   and Vocabulary only gets the Speaking swap (first weak skill wins). Vocabulary
   warmup (`phrase_match`) is never varied.
2. **No repetition avoidance.** If `email_reply` ran yesterday and today, both
   sessions are identical in shape — same pattern, same slot, every day.
3. **CEFR not used for pattern routing.** All students on a given duration get the
   same template skeleton regardless of level — CEFR only shapes AI content
   *within* a pattern, not pattern choice.
4. **Single fixed slot per `ExerciseKind`.** Templates hardcode one `PatternKey`
   per `Kind` (e.g. `ContextInput` is always `listen_and_answer` after Listening
   substitution) — no pool of alternatives to rotate through (e.g.
   `listen_and_gap_fill` vs `listen_and_answer` for Listening).

## Recommended approach (incremental)

This is **not** a quick win like the P2/P2b tasks — it's a genuine algorithm
change to `SessionGeneratorService`. Recommend breaking into 3 steps, each
independently shippable:

### Step 1 — Per-slot pattern pools + repetition avoidance (medium)

For each `ExerciseKind` in a template, define a small pool of candidate
`PatternKey`s (e.g. `ContextInput`: `listen_and_answer` | `listen_and_gap_fill`;
`VocabularyWarmup`: `phrase_match` | `gap_fill_workplace_phrase`). When generating
a session, query the student's last N `SessionExercise.ExercisePatternKey` values
(via `LearningSession` → `SessionExercise`, ordered by `CreatedAt` desc) and pick
the pool member used least recently. Falls back to template default if no history.

- Touches: `SessionDurationTemplates` (pools instead of single keys),
  `SessionGeneratorService` (new selection step + DB query for recent patterns).
- No new entities/migrations — `SessionExercise.ExercisePatternKey` already
  persisted.

### Step 2 — Multi-skill weighted substitution (medium)

Replace "first weak skill wins, one substitution" with: for each step in the
template, if `step.PrimarySkill`'s `StudentSkillProfile.ScorePercent` is the
lowest among the student's skills AND a pool alternative exists that targets that
skill, prefer it. This lets multiple weak skills each influence a different slot,
rather than one skill dominating the whole session.

- Touches: `ApplyWeakSkillSubstitution` (or its replacement), needs
  `StudentSkillProfile` scores for *all* skills (already queryable, just need to
  drop the `< WeakThreshold` filter and instead rank).

### Step 3 — CEFR-aware template selection (small, optional)

Lowest priority — CEFR already shapes AI content per-pattern reasonably well.
Could add a CEFR-based estimated-minutes adjustment (e.g. A2 students get fewer
steps / more warmup-heavy templates) but this needs product input on what "CEFR
should change the lesson shape" concretely means before scoping further.

## Sizing

- Step 1: medium — new pools data + one DB query + selection logic, isolated to
  `SessionGeneratorService`/`SessionDurationTemplates`. No migration.
- Step 2: medium — reworks `ApplyWeakSkillSubstitution`, needs care with existing
  unit tests (`SessionGeneratorServiceTests` likely exist and assert current
  single-substitution behavior — must check before changing).
- Step 3: needs product decision before sizing further — deferred.

## Decisions made

- None yet — this is scoping only. Recommend Step 1 (repetition avoidance) as the
  first implementation slice: it's the most clearly "missing" piece (zero
  variation today) and doesn't require reworking the weak-skill logic that Step 2
  would touch.

## Risks / unresolved questions

- Need to check `tests/LinguaCoach.UnitTests/Sessions/` for existing
  `SessionGeneratorService`/`ApplyWeakSkillSubstitution` tests before Step 2 —
  changing substitution logic will likely require updating fixed-expectation
  tests.
- Step 1 pool sizes should stay small (2 alternatives per slot) to keep template
  authoring manageable — confirm with product if richer pools are wanted later.
- Step 3 has no clear acceptance criteria yet — needs a product conversation on
  what "CEFR changes lesson shape" means before any sizing.

## Final verdict

Dynamic Pattern Selection is a real algorithm change, not a quick win. Recommend
starting with Step 1 (per-slot pattern pools + last-N-session repetition
avoidance) as a self-contained, testable first slice. Step 2 (multi-skill weighted
substitution) is a natural follow-up once Step 1 lands. Step 3 (CEFR-aware
shaping) needs product input before scoping.

## Next recommended action

Implement Step 1 (pattern pools + repetition avoidance in
`SessionGeneratorService`/`SessionDurationTemplates`), with unit test coverage for
the new selection logic, as the first slice of this feature.
