# Quartz JobDataMap Non-String Values — Engineering Review

Date: 2026-06-12
Related sprint: `docs/sprints/current-sprint.md`
Triggered by: production log showing `Quartz.JobPersistenceException` when
clicking "Generate lessons now" in `/admin/integrations`, plus report that all
lessons are generated on-the-fly when a student opens a module (instead of
being pre-generated by background jobs).

## Files reviewed

- `src/LinguaCoach.Infrastructure/Jobs/LessonBatchGenerationJob.cs`
- `src/LinguaCoach.Api/Quartz/QuartzConfiguration.cs`
- `src/LinguaCoach.Infrastructure/Jobs/TtsAudioGenerationJob.cs`
- `src/LinguaCoach.Infrastructure/Jobs/ActivityMaterializationJob.cs`
- `src/LinguaCoach.Api/Controllers/AdminGenerationController.cs`

## Findings

### Priority 1 — root cause: JobDataMap requires string values under `UseProperties`

`QuartzConfiguration.cs:37` sets `store.UseProperties = true` for the persistent
`AdoJobStore` (PostgreSQL-backed job store). Quartz's `useProperties` mode
serializes `JobDataMap` as a Java-properties-style key/value file, which requires
**every value to be a string** — non-string values throw
`Quartz.JobPersistenceException: JobDataMap values must be strings when the
'useProperties' property is set`.

`LessonBatchGenerationJob.TriggerAsync` (line 57-58, old code) stored:

```csharp
.UsingJobData(TriggerReasonKey, (int)reason)        // int, not string
.UsingJobData(RequestedCountKey, requestedCount)    // int, not string
```

Both calls throw `JobPersistenceException` at `scheduler.ScheduleJob(...)`,
which propagates as an unhandled exception:

- Admin endpoint: `POST /admin/students/{id}/generate-lessons` returned `200`
  from the request-logging middleware's perspective at the routing layer, but
  the actual handler throws before completing — `GlobalExceptionMiddleware`
  caught it. (The "200" log line in the report is the *start* of the request;
  the `fail` line is the actual outcome.) Either way, the admin saw no batch
  queued.
- **Any other code path that calls `LessonBatchGenerationJob.TriggerAsync`** —
  i.e. the normal background buffer-refill jobs (`LessonBufferRefillJob` and
  similar) — hits the **same exception** every time it tries to schedule a
  batch-generation job. This means scheduled/background pre-generation of
  lessons has been silently failing in production (caught/logged wherever those
  triggers are fired), which is why students only ever see lessons generated
  "on the fly" ("Preparing your lesson...") when they open a module — the
  buffer-refill path that should pre-populate `ReadyLessonBuffer` never
  succeeds in scheduling its generation job.

`TtsAudioGenerationJob.cs:43` and `ActivityMaterializationJob.cs:56` already
correctly call `.ToString()` on their job data values — only
`LessonBatchGenerationJob` had this bug, but it's the central job that both the
admin button and the background refill jobs depend on.

## Fix implemented

`src/LinguaCoach.Infrastructure/Jobs/LessonBatchGenerationJob.cs`:

- `TriggerAsync`: `TriggerReasonKey` and `RequestedCountKey` are now stored via
  `.ToString()` (matching `StudentProfileIdKey`'s existing pattern and the
  pattern already used by `TtsAudioGenerationJob`/`ActivityMaterializationJob`).
- `Execute`: reads both values back via `data.GetString(...)` +
  `int.Parse(...)` instead of `data.GetInt(...)`.

## Tests

- `dotnet build -c Release`: succeeded, 0 errors (pre-existing obsolete-API
  warnings only, unrelated).
- Existing unit/integration tests don't cover Quartz `AdoJobStore` persistence
  directly (tests use in-memory job store or don't exercise `TriggerAsync`
  against a real Postgres-backed `AdoJobStore`), so this regression wasn't
  caught by the suite. Flagged below as a follow-up.

## Decisions made

- Use `.ToString()` / `int.Parse()` rather than switching `UseProperties` off,
  since `UseProperties = true` is required for the chosen Postgres job store
  setup and other jobs already conform to the string-only constraint.

## Risks / unresolved questions

- **No test coverage for `AdoJobStore` + `UseProperties` constraints.** This bug
  could recur if a new job is added with non-string `JobDataMap` values and only
  tested against an in-memory store. Recommend an integration test that
  schedules `LessonBatchGenerationJob.TriggerAsync` against a real
  `AdoJobStore`-backed scheduler (or at minimum a unit test asserting all
  `UsingJobData` values passed are strings).
- This fix unblocks both the admin "Generate lessons now" button AND the
  background buffer-refill pipeline. Once deployed, background jobs should
  start successfully pre-generating lessons — "Preparing your lesson..." on
  module open should become rare (only for students whose buffer hasn't caught
  up yet).

## Final verdict

Root cause fixed: `LessonBatchGenerationJob.TriggerAsync` now stores
`JobDataMap` values as strings, compatible with Quartz's `UseProperties = true`
Postgres job store. This was blocking both the admin manual-generation button
and the scheduled background lesson-buffer refill jobs.

## Next recommended action

After deploy, monitor `LessonGenerationBatches` / "Recent batches" in
`/admin/integrations` to confirm batches now complete successfully, and
consider adding the test coverage noted above.
