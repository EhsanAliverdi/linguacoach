## Summary

Phase 16I implements a configurable, conservative integration path from speaking evaluation dry-run signals into real student learning state. Signal application is disabled by default (`ApplyMasterySignals: false`). Strict safety invariants are enforced: CEFR is never updated, objective completion is never triggered, and only high-confidence signals pass through.

## Findings by Priority

### P0 — Fixed during implementation

**SQLite in-memory connection not kept open (unit tests)**
- Root cause: `UseSqlite("DataSource=:memory:")` creates a per-connection database. Without `OpenConnection()` before `EnsureCreated()`, the schema was lost between EF Core connection open/close cycles.
- Fix: Added `_db.Database.OpenConnection()` in the test constructor and `_db.Database.CloseConnection()` in `Dispose()`, matching the pattern used by existing unit tests.

**StudentProfile FK violation swallowed by try-catch (unit tests)**
- Root cause: `SpeakingEvaluationSignalApplicationService.ProcessOneAsync` wraps everything in try-catch and increments `Failed`. The `StudentSkillProfile` entity has a FK to `StudentProfile`. Unit test helpers were creating evaluations with random `studentProfileId` Guids that had no corresponding `StudentProfile` row, causing `SaveChangesAsync` to fail silently.
- Fix: Both `MakeCompletedReviewEval` and `MakeCompletedPositiveEval` helpers now seed a `StudentProfile` using `student.Id` (auto-generated by `BaseEntity`) as the `studentProfileId` for the evaluation.

**Note on `StudentProfile(Guid userId)`:** The constructor parameter is `userId`, which sets `UserId`. The entity `Id` is auto-generated from `BaseEntity`. Using `student.Id` (not the constructor argument) as `studentProfileId` is correct.

**Integration test cross-contamination (IClassFixture shared DB)**
- Root cause: `RunSignalApplicationAsync` applies signals for all pending evaluations globally. Tests sharing the same `IClassFixture` DB instance contaminated each other — signals from earlier tests caused later tests to fail their `AnyAsync()` / `FirstOrDefaultAsync()` / `CountAsync()` assertions.
- Fix: Each test now queries `SpeakingEvaluations` by `LearningActivityId == activityId` to obtain the specific `EvaluationId`, then filters all `SpeakingEvaluationAppliedSignals` assertions by that `EvaluationId`.

**Idempotency test expectation wrong**
- Root cause: Test expected `result2.DuplicateSkipped == 1`. The service filters already-applied evaluations at query time via the correlated subquery, so the second run processes zero evaluations. `DuplicateSkipped` is only incremented when an evaluation passes the outer query but fails the inner idempotency check.
- Fix: Changed assertion to `result2.DuplicateSkipped == 0` with clarifying comment.

### P1 — Design decisions confirmed

**Config gate defaults to off.** `ApplyMasterySignals: false` in `appsettings.json`. Requires explicit operator action to enable.

**CEFR and objective completion permanently off in Phase 16I.** `AllowObjectiveCompletion` and `AllowCefrUpdate` are computed properties that always return `false`. Cannot be enabled via configuration.

**Positive signals gated separately.** `AllowPositiveSignals: false` by default. Review signals (`AllowReviewSignals: true`) are the only enabled signal type by default.

**Confidence threshold defaults to High.** `MinimumConfidenceForMasterySignal: "High"`. Only evaluations with all score dimensions + feedback text qualify.

**Idempotency at two levels.** Unique DB index on `evaluation_id` in `speaking_evaluation_applied_signals` provides hard idempotency. Correlated subquery in `ApplyPendingSignalsAsync` provides query-level idempotency (more efficient).

### P2 — Observations

**`HasDefaultValueSql("now()")` on SQLite.** The `SpeakingEvaluationAppliedSignalConfiguration` uses `HasDefaultValueSql("now()")` for `created_at`. SQLite does not evaluate `now()` as a function at runtime; it stores the literal. This is acceptable because `AppliedAtUtc` (the authoritative timestamp) is set in application code via `DateTime.UtcNow`. The `created_at` default is only relevant for PostgreSQL production.

**`sealed` removed from `SpeakingEvaluationProviderTestFactory`.** Required so `SpeakingEvaluationSignalApplicationTestFactory` and the pre-existing `SpeakingEvaluationQualityTestFactory` can both inherit from it. No other impact.

## Decisions Made

| Decision | Rationale |
|---|---|
| Default config: `ApplyMasterySignals = false` | Conservative rollout; operator must explicitly enable |
| CEFR always off (computed property) | Invariant too important to be config-toggleable in Phase 16I |
| Correlated subquery for pending filter | Avoids loading all applied signal IDs into memory |
| Job runs every 10 minutes | Independent of evaluation job (5 min); gives slight lag before signal application |
| Rule version `"16I-v1"` in audit record | Enables future schema-level audit of which rule logic produced each signal |
| Integration test: filter by activityId | Avoids requiring DB reset between shared-fixture tests |

## Invariants Verified

- CEFR level: never changed by speaking AI in Phase 16I (options compute `AllowCefrUpdate = false`)
- Objective completion: never triggered (options compute `AllowObjectiveCompletion = false`)
- Learning plan regeneration: service has no dependency on `ILearningPlanService`
- Failed evaluations: excluded by `Status == Completed` query predicate
- NotSupported evaluations: excluded by same predicate
- Low-confidence signals: blocked by `SpeakingDryRunConfidenceBand` integer comparison
- Duplicate application: blocked at query level (correlated subquery) and at DB level (unique index)

## Risks / Unresolved Questions

- **Frontend UI for Phase 16I**: No student-facing UI change was made. Admin endpoint (`GET /api/admin/speaking-evaluation/applied-signals`) and Angular service method are wired but no admin UI component was built. This is acceptable scope for 16I — admin can query the API directly.
- **Quartz job scheduling**: Signal application job runs every 10 minutes. Under high load with many pending evaluations, `maxBatch: 20` may create a backlog. Can be tuned via config in a later phase.
- **`HasDefaultValueSql("now()")` on SQLite**: Non-functional for SQLite in-memory tests but correct for PostgreSQL production. No production risk.
