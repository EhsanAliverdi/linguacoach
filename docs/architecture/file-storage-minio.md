# File Storage / MinIO Architecture

## Problem

Current audio storage writes files to a local filesystem path configured via `TTS_AUDIO_STORAGE_PATH`.

This is acceptable for development but is not production-ready because:
- local paths are not portable across containers or VPS restarts
- no content-addressed keys, no bucket isolation, no access control at the storage layer
- no path for scaling to object storage (S3-compatible, MinIO, Azure Blob, GCS)
- different storage approaches for TTS audio vs speaking recordings creates inconsistency

---

## Solution: `IFileStorageService`

All file storage goes through a single abstraction: `IFileStorageService`.

The backend never exposes raw filesystem paths or bucket paths to the frontend.

### Interface

```csharp
public interface IFileStorageService
{
    Task<string> SaveAsync(
        string key,
        Stream content,
        string contentType,
        CancellationToken cancellationToken = default);

    Task<Stream> ReadAsync(
        string key,
        CancellationToken cancellationToken = default);

    Task DeleteAsync(
        string key,
        CancellationToken cancellationToken = default);

    Task<bool> ExistsAsync(
        string key,
        CancellationToken cancellationToken = default);

    string GenerateKey(string ownerId, string category, string extension);
}
```

### Key Design

Keys are opaque strings generated by the backend. Format:

```
{category}/{ownerId}/{uuid}.{ext}
```

Examples:
```
tts-audio/student-abc123/f3a1b2c4-....wav
speaking-recordings/student-abc123/a7d8e9f0-....webm
```

Keys are stored in the database (`ActivityAttempt.AudioStorageKey`, listening content JSON).

The frontend never receives a key directly. It calls an authenticated backend endpoint which streams the file.

## Implementations

### `LocalFileStorageService`

Used in development and automated tests.

- Writes to a configured base directory (`FileStorage:LocalBasePath`)
- Reads files by constructing path from base + key
- `GenerateKey` produces a safe relative path
- All I/O is synchronous-wrapped as async

Configuration:
```
FileStorage:Provider = Local
FileStorage:LocalBasePath = /app/audio   (or configured path)
```

### `MinioFileStorageService`

Used in production and staging.

- Connects to a MinIO (or S3-compatible) bucket
- Uses the MinIO .NET SDK (`Minio`)
- `SaveAsync` → `PutObjectAsync`
- `ReadAsync` → `GetObjectAsync` returning a stream
- `DeleteAsync` → `RemoveObjectAsync`
- `ExistsAsync` → `StatObjectAsync`
- `GenerateKey` produces the object name

Configuration:
```
FileStorage:Provider = Minio
FileStorage:Minio:Endpoint = minio:9000
FileStorage:Minio:AccessKey = ...   (from secrets / env)
FileStorage:Minio:SecretKey = ...   (from secrets / env)
FileStorage:Minio:BucketName = speakpath-audio
FileStorage:Minio:UseSSL = false    (true in production behind HTTPS)
```

Secrets (`AccessKey`, `SecretKey`) must never be committed. They come from environment variables or Docker secrets.

## Registration

```csharp
// In Infrastructure DI registration:
var provider = config["FileStorage:Provider"];
if (provider == "Minio")
    services.AddSingleton<IFileStorageService, MinioFileStorageService>();
else
    services.AddSingleton<IFileStorageService, LocalFileStorageService>();
```

## Authenticated Streaming Endpoint Pattern

The existing pattern for TTS audio is the reference implementation:

```
GET /api/activity/{activityId}/audio
```

This endpoint:
1. Verifies student authentication
2. Verifies student owns the activity
3. Resolves storage key from activity content
4. Calls `IFileStorageService.ReadAsync(key)`
5. Streams content with correct `Content-Type`

The same pattern applies for:
- speaking attempt audio: `GET /api/activity/{id}/attempts/{attemptId}/audio`
- future uploaded files

The frontend receives the API URL, not the storage key or bucket path.

## File Categories

| Category | Key prefix | Who creates | Endpoint |
|---|---|---|---|
| TTS / listening audio | `tts-audio/` | Backend (ListeningAudioService) | `/api/activity/{id}/audio` |
| Speaking recordings | `speaking-recordings/` | Backend (SpeakingAudioService) | `/api/activity/{id}/attempts/{attemptId}/audio` |
| Future: uploaded content | `uploads/` | Backend (upload handler) | TBD |

## File Size Validation

Size limits are enforced at the API layer before calling `IFileStorageService`.

| Category | Limit |
|---|---|
| Speaking recording | 10 MB |
| TTS audio | 5 MB |
| Uploaded files (future) | 20 MB |

Validation is done in the controller/handler before persisting. Return 413 if exceeded.

---

## Content Type Handling

The storage service saves the content type as metadata where supported (MinIO object metadata).

Local storage does not persist metadata — content type is re-derived from file extension.

The streaming endpoint must set the correct `Content-Type` response header.

---

## Ownership

Every file key includes the owner's student profile ID. Backend handlers verify ownership before streaming.

Ownership check: the `ActivityAttempt` or `LearningActivity` linked to the key must belong to the requesting student. If not, return 403.

---

## Audio Cleanup

Audio cleanup (deleting old recordings to free storage) is not implemented in this sprint.

Two future options:
1. Background job in `LinguaCoach.Worker` — runs nightly, deletes audio for attempts older than N days
2. Per-student limit enforced at upload time — current approach for speaking: 50 stored attempts (DB count)

The DB count limit approach can be extended to TTS audio in a future sprint.

Cleanup must call `IFileStorageService.DeleteAsync(key)` — never delete by filesystem path.

---

## Migration Plan

Phase 5 of the implementation plan migrates existing local audio storage to the `IFileStorageService` abstraction:

1. Introduce `IFileStorageService` interface and both implementations
2. Replace `ListeningAudioService` direct filesystem writes with `IFileStorageService.SaveAsync`
3. Replace `SpeakingAudioService` direct filesystem writes with `IFileStorageService.SaveAsync`
4. Update streaming endpoints to use `IFileStorageService.ReadAsync`
5. Test with `LocalFileStorageService` locally
6. Configure `MinioFileStorageService` in staging Docker Compose
7. Verify audio playback end-to-end in staging

## Docker Compose (staging / production)

```yaml
minio:
  image: minio/minio:latest
  command: server /data --console-address ":9001"
  environment:
    MINIO_ROOT_USER: ${MINIO_ACCESS_KEY}
    MINIO_ROOT_PASSWORD: ${MINIO_SECRET_KEY}
  volumes:
    - minio-data:/data
  ports:
    - "9000:9000"
    - "9001:9001"

volumes:
  minio-data:
```

The API container connects to MinIO via the `minio` service name internally.

The MinIO console (port 9001) should not be exposed to the public internet in production.
