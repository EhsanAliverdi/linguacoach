## TODO-1: Playwright E2E test suite

**What:** Write Playwright E2E tests for the full browser-level user flows.

**Why:** Unit and integration tests cover API correctness, but don't test the actual browser experience: token refresh happening silently in the background, onboarding resume after a real page reload, the full login → onboarding → dashboard flow as a real user would experience it.

**Pros:** Catches browser-level bugs that unit tests miss. First real user session validated by automation before it happens.

**Cons:** Playwright setup is non-trivial. Requires a running app instance (either local or staging). Adds ~1 day to scope.

**Context:** Playwright config is added in slice 1 (`frontend/linguacoach-app/playwright.config.ts`) but no tests are written. Two E2E-worthy flows identified in the test review:
1. Full login → onboarding (all 4 steps) → dashboard flow
2. Token refresh: token expires mid-session, app silently renews without user seeing a login screen

**Depends on:** T12 (CI/CD) — E2E tests need a stable deployment target to run against.

## TODO-4: Memory update staleness detection

**What:** If `UserLearningSummary.UpdatedAt` is stale (> 7 days old) but the student has recent `ActivityAttempt` rows, surface an admin alert or trigger a background refresh.

**Why:** The memory update is best-effort and silently swallows failures. If the AI provider is down for hours or a bug causes repeated failures, the student's memory drifts silently — adaptive generation degrades without anyone knowing.

**Pros:** Makes the silent failure visible. Admin can see which students have stale memory. Auto-retry gives the system a chance to self-heal.

**Cons:** Requires a background job or scheduled check in `LinguaCoach.Worker`. Admin alert UI is new surface area.

**Context:** Added during Student Learning Memory sprint (2026-06-05). Currently `StudentMemoryService.UpdateMemoryAsync()` logs a warning with correlation ID on failure but takes no further action. Staleness threshold suggestion: 7 days with recent activity = stale. Check can run as a daily background task.

**Depends on:** Student Learning Memory sprint (memory update feature) must be shipped first.

## TODO-6: Speaking audio cleanup job

**What:** Background job (hosted service or Hangfire task) that deletes speaking audio files for attempts older than 90 days, or when a student's stored file count exceeds a soft ceiling.

**Why:** The SpeakingRolePlay sprint adds per-student audio storage with a DB-count guard (50 files max). Without a cleanup job, old files accumulate indefinitely and the hard limit eventually blocks students with no self-service recovery.

**Pros:** Prevents unbounded disk growth. Removes the hard 50-file ceiling as a student-blocking issue.

**Cons:** Requires a background job infrastructure decision (Hangfire vs .NET hosted service vs CLI). Adds configuration for retention period. Needs soft-delete vs hard-delete decision for compliance.

**Context:** Added during SpeakingRolePlay MVP review (2026-06-08). The DB count guard (TODO deferred cleanup) is the short-term mitigation. This job is the permanent fix. Retention window suggestion: 90 days. Files are identified by `ActivityAttempt.AudioStorageKey` — a count query against the DB gives the list to clean.

**Depends on:** SpeakingRolePlay MVP sprint must ship first.

## TODO-7: Real STT provider (OpenAI Whisper or Azure Speech-to-Text)

**What:** Implement a production `ISpeechToTextService` backed by a real speech-to-text API (OpenAI Whisper recommended; Azure Speech-to-Text as alternative).

**Why:** The MVP uses `FakeSpeechToTextService`, which returns the same placeholder transcript regardless of what the student said. This means AI evaluation feedback is always identical. Speaking practice only becomes genuinely useful when transcripts reflect real student recordings.

**Pros:** Speaking feedback becomes personalised to the student's actual words. Strengths/improvements are real. Students get genuine value from the activity.

**Cons:** Adds a paid API dependency. Requires secrets management (API key). Audio bytes are sent to a third party — privacy and data residency implications for the target cohort (immigrant professionals in Australia).

**Context:** Added during SpeakingRolePlay MVP review (2026-06-08). Fake STT is sufficient for the pilot to validate the UI flow and feedback structure. Switch to real STT when the pilot produces real recordings and feedback quality matters. OpenAI Whisper is recommended (better multilingual support for students with Persian-accented English). Check Australian data residency requirements before choosing provider.

**Depends on:** SpeakingRolePlay MVP sprint must ship first. Consider data privacy review before choosing provider.

## TODO-8: Configurable Learning Session Templates

**Added:** 2026-06-10
**Related sprint:** Today's Lesson / Learning Session

**What:** Move lesson session templates from hardcoded `SessionDurationTemplates.cs` into seeded database records and eventually an Admin UI.

**Why:** The current code-owned templates are acceptable for MVP. They are deterministic, fast, and require no DB reads during generation. However, as the product matures the curriculum team will need to adjust step sequences, add new exercise patterns, and tune durations without a code deploy. Keeping templates in code limits that flexibility.

**Future capability:**
- Admin can create, edit, and disable lesson templates
- Templates are bucketed by duration: 10, 15, 20, 30, or custom minutes
- Each template record has ordered step definitions
- Each step defines: `ExerciseKind`, title, estimated minutes, required/optional flag, target `ActivityType`, skill focus, prompt key, and metadata JSON
- One template can be marked as the default for its duration bucket
- Templates are versioned so that sessions already generated from an older template are not broken when the template is edited
- An Admin preview/test mode shows the generated lesson structure for a given student profile without actually creating a session
- The generator (`SessionGeneratorService`) continues to own all safety rules: weak-skill substitution, domain complexity caps, idempotency. Templates supply the scaffold; the generator applies student-specific overrides on top.

**Why deferred:**
- Code templates are sufficient for the pilot cohort
- Admin UI is not in scope for the Today's Lesson MVP
- Versioning and preview mode add significant scope
- The generator is already structured to accept a step-list from any source — swapping from static data to DB records is a contained change when the time comes

**Migration path when ready:**
1. Add `lesson_session_templates` and `lesson_session_template_steps` tables (new migration)
2. Seed the current hardcoded templates as DB records
3. Update `SessionGeneratorService` to load the active template from DB instead of the static class
4. Add Admin UI pages for template management
5. Add template preview endpoint

**Depends on:** Today's Lesson MVP sprint must ship first. Admin UI baseline (curriculum management) must be unblocked first.

## TODO-9: GenerationBatch SummarySnapshotJson cleanup policy

**Added:** 2026-06-11
**Related sprint:** Lesson Buffer / MinIO / Background Generation

**What:** Add a background job (or extend AudioCleanupJob) that prunes old `GenerationBatch` rows — or at minimum truncates the `SummarySnapshotJson` column — after a retention window.

**Why:** Each `GenerationBatch` row stores the full compact summary JSON (~1–2KB). At 4 sessions per batch × ongoing generation × N students, this grows indefinitely with no cleanup policy.

**Pros:** Bounds table size. Prevents slow queries on `GenerationBatch` as row count grows.

**Cons:** Must not delete rows still needed for audit or retry. Requires a soft-delete or archive strategy.

**Context:** Added during eng review of Lesson Buffer sprint (2026-06-11). At early scale (< 100 students) this won't matter for months. Suggested retention: keep last 10 batches per student, archive or delete older rows. The `SummarySnapshotJson` is advisory — it's stored for auditability and replay, not for active operation.

**Depends on:** Lesson Buffer sprint must ship first (GenerationBatch table must exist).

## TODO-10: DB role default privileges — app runtime role missing grants on new tables

**Added:** 2026-07-04
**Related:** Unified JSON Question-Schema Phase 6b (`13513ec`), see `docs/reviews/2026-07-04-onboarding-categories-placement-permission-500-debugging.md`

**What:** Production Postgres has no `ALTER DEFAULT PRIVILEGES` configured for the `linguacoach` role (the role migrations run/create tables as). The app's runtime role, `linguacoach_dba`, only has explicit grants on tables that existed at some earlier point — every table created by a migration since then (`onboarding_category_definitions`, `placement_item_definitions`, confirmed; possibly others from earlier this session) has zero grants to `linguacoach_dba`, causing silent permission-denied 500s the first time a code path touches that table.

**Why:** This already caused two live production 500s (`/api/admin/onboarding/flow`, `/api/admin/placement-items`) after the Phase 6b deploy, and will recur on every future migration that adds a table unless fixed at the role level.

**Fix (blocked on credentials — `linguacoach_dba` cannot grant on tables it doesn't own):**
```sql
GRANT SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER
  ON onboarding_category_definitions, placement_item_definitions
  TO linguacoach_dba;

ALTER DEFAULT PRIVILEGES FOR ROLE linguacoach
  GRANT SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER
  ON TABLES TO linguacoach_dba;
```
Must be run by the `linguacoach` owner role or a Postgres superuser.

**Depends on:** User supplying owner/superuser DB credentials, or running the SQL directly via their own DB console.

## TODO-11: "Answer Short Question" speaking pattern loses all answers via typed fallback

**Added:** 2026-07-08
**Related:** `docs/testing/2026-07-08-full-app-qa-bug-bash-report.md` (finding D1)

**What:** When microphone access is unavailable (denied, non-HTTPS, no hardware — a common real-world case), the "Answer Short Question" 5-item speaking practice falls back to a single "Type what you would say…" textbox instead of the per-item `AnswerShortQuestionComponent` (which properly tracks one response per question and emits `{itemId, answerText}[]`). The single typed answer never reaches the 5 expected answer slots, so feedback shows all 5 questions as "(no answer)" / 0% correct regardless of what was typed.

**Why:** Two parallel implementations of the same speaking-pattern concept appear to exist — the legacy single-response voice-recorder/teach-page flow (`activity-teach-page.component.html`'s `case('speakingScenario')`) and the newer per-item pattern-engine renderer (`exercise-renderer.component.ts` + `AnswerShortQuestionComponent`). This activity instance resolved to the legacy flow instead of the correct multi-item one.

**Fix (needs investigation before implementing):** Trace the routing/dispatch decision that picks `activity-teach-page`'s `speakingScenario` case vs. `exercise-renderer`'s pattern-engine path for `answer_short_question`-keyed activities, and correct it so multi-item speaking patterns always resolve to the per-item renderer (with its own working mic-denied fallback, if one exists, or add one).

**Depends on:** Nothing blocking — can be picked up directly.
