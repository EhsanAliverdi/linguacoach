## TODO-1: Playwright E2E test suite

**What:** Write Playwright E2E tests for the full browser-level user flows.

**Why:** Unit and integration tests cover API correctness, but don't test the actual browser experience: token refresh happening silently in the background, onboarding resume after a real page reload, the full login → onboarding → dashboard flow as a real user would experience it.

**Pros:** Catches browser-level bugs that unit tests miss. First real user session validated by automation before it happens.

**Cons:** Playwright setup is non-trivial. Requires a running app instance (either local or staging). Adds ~1 day to scope.

**Context:** Playwright config is added in slice 1 (`frontend/linguacoach-app/playwright.config.ts`) but no tests are written. Two E2E-worthy flows identified in the test review:
1. Full login → onboarding (all 4 steps) → dashboard flow
2. Token refresh: token expires mid-session, app silently renews without user seeing a login screen

**Depends on:** T12 (CI/CD) — E2E tests need a stable deployment target to run against.

## TODO-2: Admin CRUD for career profiles and learning tracks

**What:** API endpoints and Angular UI for admin to create/edit/deactivate career profiles and learning tracks without database intervention.

**Why:** The product requirement says "admin manages career profiles." Slice 1 seeds one career profile (Document Controller) and one track (Workplace English). Adding a second career profile (e.g. Site Engineer, Nurse) currently requires a developer to insert database records directly.

**Pros:** Makes the product self-serve. Admin can onboard new student cohorts with different roles without developer involvement. Required before the product can scale beyond the first pilot.

**Cons:** Significant scope — admin CRUD UI + validation + API. Not needed until the second career profile is confirmed.

**Context:** The `LearningTrack` and `CareerProfile` entities exist in slice 1 with their schemas. `GET /api/reference/tracks` and `GET /api/reference/career-profiles` already serve them dynamically. What's missing is the admin write path: `POST /api/admin/career-profiles`, `PUT /api/admin/career-profiles/:id`, etc.

**Depends on:** Slice 1 must be deployed and stable before admin UI is worth building.

## TODO-3: Email sending — automated temp password delivery

**What:** Implement the email stub in `LinguaCoach.Infrastructure` to actually send emails. First use case: when admin creates a student, system auto-emails the student their temp password and login URL.

**Why:** Current flow is out-of-band (admin tells the student their temp password manually via Slack, WhatsApp, etc.). This works for 5 students. It doesn't scale beyond 10.

**Pros:** Removes admin friction. Student gets an email with clear instructions. More professional first impression for the product.

**Cons:** Requires choosing and integrating an email provider (SendGrid, Postmark, AWS SES). Adds secrets management. Non-trivial to test end-to-end.

**Context:** `LinguaCoach.Infrastructure` has an `IEmailService` stub from slice 1. The `POST /api/admin/students` endpoint returns the temp password in the response body (shown once to admin). The email feature replaces that manual step. Suggested provider: Postmark (strong deliverability, good .NET SDK).

**Depends on:** TODO-2 is not required. Can be built independently once slice 1 is deployed.
